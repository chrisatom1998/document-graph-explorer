API RATE LIMITING

Every request that reaches api-gateway is subject to rate limiting before
it's routed anywhere else. This document covers how limits are set, how a
client experiences being throttled, and how rate limiting interacts with
both the Kubernetes layer and the auth layer -- it sits at the seam
between architecture-overview.md, auth-service.md, and
kubernetes-conventions.md, which is why it doesn't live cleanly under any
one team's usual doc set.

WHY RATE LIMITING EXISTS HERE SPECIFICALLY

Two failure modes drove this: a misbehaving internal client hammering the
JWT token endpoint on auth-service hard enough to look like an attack, and
an external customer integration retrying failed requests in a tight loop
against kafka-event-bus-backed endpoints faster than consumers could
drain the resulting load. Rate limiting is the shared circuit breaker for
both cases.

HOW LIMITS ARE SET

Limits are scoped per API key (external clients) or per service identity
(internal calls, using the same JWT scope claims described in
auth-service.md). Default limits are generous for normal usage and tight
specifically around the token issuance endpoint, since that's the
cheapest request to send and the most expensive to handle at scale.
Limits are configured as part of each service's Kubernetes deployment
(see kubernetes-conventions.md) rather than hardcoded, so they can be
tuned without a full redeploy.

WHAT A CLIENT SEES

A throttled request gets an HTTP 429 with a Retry-After header. Well
-behaved clients back off and retry after that window; clients that
ignore the header and retry immediately get increasingly aggressive
throttling, the same way a Kafka producer that ignores backpressure just
makes its own problem worse, as described in kafka-event-bus.md.

INTERACTION WITH BLUE-GREEN DEPLOYS

Rate limits are enforced identically against both the blue and green
fleets during a rollout (see deploy-guide.md) -- a client should never be
able to bypass its limit just because a deploy happens to be mid-rollout.
This was a bug once, briefly, and fixed within a day; it is not a bug now
but it's worth knowing it used to be.

WHEN THROTTLING LOOKS LIKE AN INCIDENT

If a legitimate customer integration suddenly starts getting throttled at
a much lower threshold than usual, check whether their traffic pattern
changed before assuming the limiter is misconfigured -- but if error rates
climb broadly across many clients at once, that's not normal throttling,
that's an incident, and belongs in the Incident Runbook, not a rate-limit
support ticket.

CHANGING A LIMIT

Raising a customer's rate limit is a routine support-driven change.
Lowering the global default, or changing anything about how the token
endpoint specifically is throttled, requires the same sign-off as any
other auth-service change, per the Security Review Process.
